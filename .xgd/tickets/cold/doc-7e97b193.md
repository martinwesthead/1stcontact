# Module Conformance: Universal Contract ACs & the Shared Test Harness

## Purpose

A module is a **contract**, not just reusable markup. Its value over hand-coded output is a machine-verified promise — *safe, secure, cross-browser, responsive* — that every site inherits. This doc defines how that promise is expressed as **capability-matrix entries** (not a CI bolt-on) and verified by a **shared conformance harness**, within the constraints of XGD's actual data model.

Companion docs: [[DOC-14]] (module lifecycle / two-tier composition), [[DOC-7]] (framework architecture), [[DOC-19]] (the faelan cross-browser evidence). System docs: `PHILOSOPHY.md` (capability matrix, evidence-first), `FRAGILE-INTENT-LIFECYCLE.md` (immune model, interface/implementation line, multi-surface gap). Related tickets: [[REQ-17]] (publish-gate-on-hardening), [[REQ-38]] (perceptual diff — doubles as the cross-engine diff engine + eyes), [[REQ-13]] (`1c shot`).

## The two AC sets on every module

Each module maps to a **story** (its intent: why it exists, what each parameter/variant does) under a "Module System" capability. Two kinds of AC hang off that story:

1. **Bespoke functional ACs** — the module's own contract (`services-grid` with N items → N cards; `hero` scrim darkens the image). Module-specific; authored from the intent body.
2. **Universal module-contract ACs** — hold for *every* module "just because it is a module": safety, security, cross-browser, responsive. Identical set across all modules.

Both are **behavioral and stable across refactors** — the AC quality bar `PHILOSOPHY.md` demands. "No horizontal overflow at width ≥320", "layout boxes match across engines within tolerance", "content fields render injection-inert" survive a total module rewrite. They are proper matrix citizens, not implementation assertions.

## Why matrix-native, not a CI folder — the immune payoff

`FRAGILE-INTENT-LIFECYCLE.md` §1: the matrix is the *definition of self*; surveillance (`test_fix` + regression) eliminates drift from it. Putting the universal ACs **in the matrix** makes conformance **self** — so cross-browser correctness, no-overflow, and injection-safety become **antibodies**. A module that later regresses (starts overflowing, breaks in WebKit, leaks an unescaped field) is caught and fixed by the immune machinery *forever*, not just at authoring time. That is the whole reason to pay the matrix cost instead of dropping spec files in a folder: conformance becomes protected self the system defends on its own.

## The tree constraint (and the pattern it forces)

**The capability matrix is a tree, not a DAG.** There is no shared AC/UAT node many module stories can inherit. Every module story must own its own leaf UATs → the universal ACs are **physically duplicated per module**. This is forced by the data model; accept it.

The resolution is the **thin-leaf / shared-harness** split, which is *blessed*, not a workaround. `FRAGILE-INTENT-LIFECYCLE.md` §3.3: *"the matrix is interface, not implementation... choices below that surface do NOT become matrix entries."*

- The **tree** gets one thin leaf per module per universal AC — the interface (pins the behavior).
- The **harness** lives once in framework source, *below* the matrix line — the implementation the leaf delegates to.

The leaf is three lines:
```ts
test('UAT_FC_<MODULE>_responsive', () =>
  assertModuleConforms('services-grid', servicesGridFixtures, { dimension: 'responsive', tier: 'fast' }))
```
Per-module hand-authored content shrinks to **fixtures + declared exceptions**; fixtures are mostly schema-derived (enumerate Zod variants + boundaries). The irreducible per-module labor is tiny.

## Template-and-stamp replaces inherit-and-share

Since the node can't be shared, do the tree-compatible equivalent: **one template, machine-stamped into every module.** The module-contract (the 4 universal ACs + their shim UATs) is authored **once** as a template; the module-authoring / reconcile flow stamps copies into each module story, filling `slug` + fixtures. The tree physically holds 4×M copies, but there is a single source of truth.

Regeneration is clean in matrix terms: changing the universal set is a **module-contract intent** that *supersedes* the prior stamped ACs across all module stories (`FRAGILE` §3.2 supersession — a fan-out, but intent-owned and well-defined, never an ad-hoc script edit). This is the tree-native substitute for DAG inheritance.

**Keep the 4 ACs distinct** (safety / security / cross-browser / responsive) rather than one opaque `conforms` blob — evidence-honesty (`proof.md` shows *which* dimension is unproven) requires the matrix to say which dimension failed.

## Cost split: fast leaf in the tree, full matrix in regression

`3 engines × 6 viewports × eyes` is far too heavy for a leaf that runs on every `test_fix`/develop cycle. Split the cost onto XGD's existing fast/thorough divide:

| Layer | Runs | Scope |
|---|---|---|
| **Thin leaf (`tier: 'fast'`)** | in the tree, every cycle | 1 engine (Chromium), mechanical checks (console/overflow/collapse/injection), 1–2 widths |
| **Full matrix (`tier: 'full'`)** | **regression** antibody, periodic, cuts from main | WebKit + Gecko layout-equivalence, full viewport sweep, eyes pass |

Cross-browser drift is a *surveillance* concern (does main still hold across engines?) — regression's job by definition. This also avoids running WebKit/Gecko per unit test. **Prerequisite:** the full tier needs the WebKit + Firefox browser binaries installed (`playwright install webkit firefox`) — a permissioned install; regression runtime rises ~3×.

## The harness interface

Shared implementation in framework source (e.g. `framework/testing/conformance`), below the matrix line:

```ts
interface ConformanceFixture { label: string; props: Record<string, unknown> }
interface ConformanceOptions {
  dimension?: 'safety' | 'security' | 'x-browser' | 'responsive'   // omit = all
  tier?: 'fast' | 'full'                                            // default 'fast'
  viewports?: number[]                                             // default per tier
  engines?: ('chromium' | 'webkit' | 'firefox')[]                 // default per tier
  except?: string[]                                               // AC ids legitimately opted out (declared + reasoned)
}
// The one call every thin leaf makes. Throws (fails the UAT) on any non-excepted violation.
async function assertModuleConforms(
  slug: string, fixtures: ConformanceFixture[], opts?: ConformanceOptions
): Promise<void>
```

**Isolation model:** render a *one-module page* through the catalog-driven renderer (realistic — includes theme + layout), serve over loopback (the same seam as `1c shot` / `values-diff`), drive with Playwright. This tests what actually ships, not a bare component mount.

**Dimensions → checks:**

- **safety** — no console/page errors; `scrollWidth ≤ viewport.width` (no horizontal overflow); no expected-content container collapsed to 0; text not clipped. + eyes ("does it look broken?") at `full` tier only (advisory).
- **security** — per content field, mount injection payloads (`<script>`, `"><img onerror>`, `javascript:`/`data:` URLs, markdown-embedded HTML) → assert inert (escaped, no execution, safe `href`/`src`); no content reaches inline `style=`; no network request outside the asset allowlist.
- **x-browser** — extract element layout boxes in each engine; assert positions/sizes match within tolerance (calibrated against the known faelan %-top Safari/FF shift — [[DOC-19]]); perceptual backstop reuses the [[REQ-38]] block-averaged diff (reference = Chromium render).
- **responsive** — run the safety checks at each of {320,375,768,1024,1280,1440}; tap-targets ≥44px on mobile; font-size floor; images scale; no overflow at ≥320.

## Who tests the harness (the highest-risk part)

If every module's leaf delegates to `assertModuleConforms`, a **buggy harness turns 4×M leaves into rubber-stamps** — the "model grading its own homework" failure, mechanized at scale. The harness therefore needs its own **test-infrastructure UATs built on negative fixtures**: deliberately-broken modules that the harness **must flag red** before delegation can be trusted.

- an overflowing module → `safety`/overflow flagged
- a module leaking an unescaped content field → `security` flagged
- a module that collapses below 480px → `responsive` flagged
- a module that diverges in WebKit (the faelan %-top gap) → `x-browser` flagged
- a clean module → **passes** (no false positive)

Prove the discriminator discriminates *first*. Non-negotiable.

## The module-contract template (universal ACs, behavioral)

- **AC-M1 (safety):** At any supported viewport, the module produces no console or page errors and no horizontal overflow; no expected-content container collapses; text is not clipped.
- **AC-M2 (security):** Every content field renders untrusted input inert — no script execution, no unsafe `href`/`src` scheme, no escape into inline styles — and the module issues no network request outside the asset allowlist.
- **AC-M3 (cross-browser):** Element layout boxes match across Blink, WebKit and Gecko within tolerance; no element shifts more than the calibrated threshold between engines.
- **AC-M4 (responsive):** From 320px up, content stays within the viewport, legible (font-size floor) and operable (tap targets ≥44px on mobile); the mobile layout requires no per-viewport authoring.
- **Exemption rule:** a module MAY opt out of a specific universal AC by declaring the AC id + a documented reason in its story (e.g. a full-bleed gallery that intentionally scrolls horizontally). The opt-out is matrix-recorded and surfaced in `proof.md`, so the immune system does not eliminate legitimate variance.

## Lifecycle integration

Maps onto [[DOC-14]] two-tier + [[REQ-17]]:
- **Draft / Tier-B (free-code):** conformance advisory — the `fast` leaf runs but doesn't block, so bespoke modules can iterate. (Matches pre-prod laxity.)
- **Harden → Tier-A (publish gate):** full conformance mandatory. This work **defines the hardening criteria** [[REQ-17]]/[[DOC-14]] currently leave abstract.

## Long-term: this is a known XGD gap, and 1stcontact is its forcing function

The DAG we can't have is really two deferred XGD concepts:
1. **Trait-ACs / AC templates** — a named, versioned AC bundle inherited by story *type*, stored once, supersession propagating to members. Kills the prose duplication.
2. **Multi-surface specification — `FRAGILE-INTENT-LIFECYCLE.md` §6, gap #6.** The current matrix line is *user behaviour*; plugin/contract surfaces "are not currently expressible as matrix entries... does not scale to library/framework projects. The likely shape: intent tickets declare their surface levels, and the matrix admits entries at any declared level."

That gap **is** this situation: a module is a plugin-contract surface, and the universal ACs are contract-level ACs. So the "new XGD concept" is not a 1stcontact bespoke — it's the resolution of gap #6 (declare a `module-contract` surface; the matrix admits a trait-AC bundle at it). **Strategically:** gendev owns XGD, 1stcontact is a gendev project, so 1stcontact needing multi-surface ACs is the real-world pressure that should drive gap #6 into XGD proper — dogfood, and the capability returns to every future framework project.

## Build slicing

Framework code → full free-coding ceremony (scope ticket + `test_UAT_FC_*` + `[FREE-CODED]` + version bump).
1. **Harness + negative fixtures + `assertModuleConforms`, Chromium-only `fast` tier.** The discriminator, proven first. Highest value, unblocks everything.
2. **Responsive axis** (viewport loop over the fast checks) — cheap.
3. **Cross-browser `full` tier** (WebKit + Gecko layout-equivalence, reuse [[REQ-38]] diff), calibrated on faelan — the expensive, highest-novelty piece; regression-scoped.
4. **Module-contract template + stamp** into module stories; wire the publish gate ([[REQ-17]]/[[DOC-14]]).

Short term: stamp the ACs per module (prose duplicated, tests shared via the harness). Long term: trait-ACs + multi-surface (gap #6) removes the prose duplication.
