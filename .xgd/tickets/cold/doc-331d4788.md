# Module Lifecycle & Two-Tier Composition

## 1. Purpose

How the framework's design language grows **without limit** while staying safe and reproducible: the two-tier composition model, and the lifecycle of a bespoke module from in-session draft to hardened, published primitive. Companion to [[DOC-7]] §6 (philosophy) and [[DOC-4]] (product thesis).

## 2. Two-tier composition

- **Tier A — compose from the library.** Structured instance data (config, layers, positions, motion / generative params) over reusable modules. Fast, AI-driven, safe — the 95% path.
- **Tier B — author a bespoke module.** When the library cannot express a design, author a new component (code) that exposes **structured params** to the instance. The structured escape valve that never leaves the model.

The module is a **structured escape valve to arbitrary code.** Template builders have *no* code escape (a hard, low ceiling); we have one that stays structured — so there is no expressive ceiling. **Instance data is always structured** ([[DOC-7]] §6.2); **module code is a gated artifact** (this doc).

## 3. What triggers a bespoke module

An **eyes-verified capability gap** during composition. The AI composes with the library, renders, screenshots, and compares to the target:

- **Config gap** → keep tuning dials/positions/params. No new module.
- **Capability gap** → *no amount of config closes this*. **Trigger.**

The trigger emits a **capability spec** — a structured request + the target screenshot. (This is the capture **gap-log** of [[DOC-13]].)

## 4. Lifecycle

```
capability gap (eyes)
 → capability spec            request + target screenshot
 → 1C, in-session             Claude drafts the module           [DRAFT — works for this site]
 → operator approves the look in draft preview                   [eyes sign-off]
 → XGD hardens                RED/GREEN tests, quality gates,
                              security review, generalize + moduleMeta contract
 → published                 site-local OR promoted to library
 → the site can publish
```

## 5. Key properties

1. **Rides draft/published ([[DOC-12]]).** Draft sites may use **draft (unhardened)** modules — that is how the operator sees the site working *now*. **Publishing gates on hardening**: a site cannot go live while it depends on an unhardened bespoke module. Draft = experimental; published = only hardened. This extends the forward-only publish gate — no new machinery.
2. **The approved draft screenshot is the hardening acceptance test.** The operator signs off on the *look* in draft; that screenshot becomes XGD's **visual-regression target**. "Hardened" means *passes tests **and** still looks like what was approved.* The eyes loop supplies the dev loop's acceptance criteria.
3. **The security line holds where we put it.** Module code — draft or hardened — is a **gated artifact**, never reaching production without XGD's security review; **instance data stays structured** ([[DOC-7]] §6.2). "AI writes code" is fine here because it is the gated Tier-B escape valve, not the instance-editing path.

## 6. Site-local vs library promotion

Decided at hardening, informed by **gap frequency** across captures:

- **Site-local** — a one-off (narrow reuse); hardened, published with that site.
- **Library primitive** — the gap recurs across sites; XGD generalizes it into a reusable module. This is how the language grows ([[DOC-7]] §6.3) — by real demand through the pipeline, not by fiat.

## 7. The flywheel

XGD is already *Claude doing rigorous TDD development*. A bespoke module is a small piece of software. So the **site-builder** (Claude in 1C) discovers a needed capability, and the **dev-system** (Claude in XGD) builds it properly and folds it into the library. The product **grows its own expressive language through its own development process** — which is what makes "no expressive ceiling" economically real, not merely theoretical.

## 8. Honest edges

- The in-session draft may be **rebuilt from the spec** during hardening (works-now vs production-grade). The draft's value is immediate site progress + a concrete reference, not final code.
- **Publish latency** — a site with a novel module waits for hardening before going live. This is the correct gate (no untested code live).
- **Render must resolve site-local / draft modules**, not just the shared library — a small extension to `1c render`'s module resolution.
