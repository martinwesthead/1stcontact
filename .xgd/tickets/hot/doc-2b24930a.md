# Crawler & Framework Coverage Program

## 1. Purpose — build the framework, not a clone service

The crawler is an **internal framework-development tool.** Its output is a **more capable framework** — the right modules and primitives, built in the right order — *not* a customer-facing "clone any site" feature and *not* a redistributable library of other people's sites.

It censuses the real design space so module development is **data-driven**: build what the market actually needs, in ROI order, and prove the expressive ceiling ([[DOC-4]]). It is the batch, offline form of the [[DOC-14]] flywheel.

The customer-facing "reproduce a specific site" experience is a **separate, deferred** product surface — see §7.

## 2. Mechanism — the offline flywheel

Crawl for breadth → **capture** ([[DOC-13]]) → **attempt reproduction** with the current library → **classify by eyes** → **histogram** → build.

Classification is a reproduction *attempt*, not static analysis — the [[DOC-14]] loop run in batch:

1. **Covered** — reproduced with existing modules; screenshot matches (eyes).
2. **Generalizable** — close, but needs a module to accept one more value / variant / param (a cheap extension).
3. **New module** — cannot get close without a genuinely new component.

Depends on the capture tool (REQ-12) and sharpens as the library grows (early on, with a tiny catalog, almost everything classifies "new").

## 3. Phases

- **Phase 1 — accumulate a diverse corpus (a few thousand).** *Coverage-driven, not volume-driven*: keep a capture only if it adds design-space coverage (dedup / cluster by capture signature). The metric is *fraction of the design space covered*, not sites captured.
- **Phase 2 — cover the bases.** Be able to reproduce **every Wix template and every out-of-the-box WordPress / Elementor / Divi theme.** This is **finite** (template galleries are enumerable) and **market-covering** (those templates generate most real small-business sites). Crawl the **galleries / demos**, not live customer properties (§6).
- **Phase 3 — histogram the tail.** Rank remaining gaps by **frequency × cost**: cheap generalizations first, new modules by frequency, true one-offs handled as **bespoke site-local modules** ([[DOC-14]]) rather than library primitives.

## 4. Two roadmap drivers

The module roadmap has two independent inputs, and they pull apart:

- **Coverage (the histogram)** → the common modules that win the *breadth* of the market.
- **The corollary (ceiling proof)** → the wow-factor primitives (motion, generative visuals, interactivity). A pure-frequency histogram would *never* prioritize these, because the small-business corpus is mostly not flashy.

**Corollary as acceptance test:** *rebuild 1stcontact.io to sycamore.so-level using only our own modules.* Passing it proves the language + eyes reach the ceiling, and it is the public demonstration [[DOC-4]] already promises. Without it we would build a very complete *boring* builder.

## 5. Output

A **prioritized module backlog** (generalizations + new modules, ROI-ordered) and a **coverage map / dashboard**. New modules flow through the [[DOC-14]] harden pipeline into the library.

## 6. IP & sourcing stance (principle, not blocker)

- **Capture learns capabilities, never redistributes content.** The corpus is **internal** and **transformative** — capability signals, not a content archive, and not a library for customer consumption.
- **Reproduction copies form; the customer supplies content.** (Form — layout, style, effects — is not protectable; content and brand are.)
- **Source Phase 2 from public template galleries / theme demos**, crawl politely (robots, rate limits), and avoid mass-crawling a platform's live *customer* sites (a Terms-of-Service, not copyright, concern).
- **Revisit with counsel** before any public, at-scale, customer-facing feature.

This keeps the program moving while staying honest; none of it blocks the internal work.

## 7. Deferred — the customer-facing "a site like X" experience

Explicitly **out of scope** and **deferred**: the "I'm a plumber, reproduce my competitor's site" / "I want a site like kylieminogue.com" experience. It is a separate surface with its own field of landmines — IP (content/brand), **illegal content**, hosting liability, and more. The internal crawler/coverage program **does not depend on it and does not build it.**

Open design question, when we do take it on:

> "A site *like* X" — reproducing **style, special effects, and structure** is probably legitimate (form / capabilities); **copyright content** (assets, copy, brand) must be stayed clear of. **Prompts and tooling must enforce the form ≠ content boundary.** Good AI can help navigate this, but it needs deliberate design and counsel before shipping.

## 8. Sequencing & dependencies

- Downstream of **REQ-12** (capture). Phase 1 can begin once capture lands; classification matures with the library; Phases 2–3 need a baseline catalog.
- The coverage engine is the **batch form of the [[DOC-14]] flywheel**; discovered modules flow through [[DOC-14]]'s draft → harden → publish lifecycle.
- Spawns future tickets (not yet filed): the crawler (breadth / novelty-driven), the batch coverage/classification engine, the corpus store + coverage histogram, and the **sycamore-grade 1stcontact.io** corollary milestone.
